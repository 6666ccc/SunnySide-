# SunnySide 优化待办清单

> 基于当前代码与配置（`src/main/java`、`src/main/resources`、`pom.xml`、`sql.sql`）整理。
> 优先级说明：`P0` 立即处理（上线前必须完成）、`P1` 重要改进、`P2` 体验与扩展增强。

## P0（上线前必须完成）

- [ ] **移除并轮换已暴露密钥与默认弱口令**
  - 目的：避免 API Key/数据库凭据泄露导致数据与账单风险。
  - 现状：`src/main/resources/application-local.yml` 中存在明文 `dashscope api-key`；`application.yml` 中有默认数据库口令。
  - 验收标准：
    - 所有密钥仅通过环境变量或密钥管理注入；仓库中无明文密钥。
    - 已完成密钥轮换（旧 key 失效）。
    - 新增 `.env.example` 或文档模板，明确必填环境变量。

- [ ] **收敛 CORS 策略，禁止生产环境全开放**
  - 目的：降低跨站请求与凭证滥用风险。
  - 现状：`src/main/java/cn/lc/sunnyside/WebConfig/WebConfig.java` 允许 `*` 来源并开启 `allowCredentials(true)`。
  - 验收标准：
    - 生产环境仅允许白名单域名（按 profile 配置）。
    - 本地开发与生产策略分离。
    - 安全评审通过（至少覆盖 cookie/token 场景）。

- [ ] **为 AI 接口补齐认证与限流**
  - 目的：防止匿名滥用模型与工具调用造成成本和安全问题。
  - 现状：`AIController` 的 `/chat`、`/stream`、`/chat/multimodal` 目前可直接访问。
  - 验收标准：
    - 接口接入统一认证（如 JWT/Spring Security）。
    - 按用户/IP 做限流与配额控制。
    - 未认证请求返回一致错误码（401/403），并有审计日志。

- [ ] **修复活动报名日期逻辑错误**
  - 目的：避免“报名日期与活动日期不一致”导致查询和业务混乱。
  - 现状：`ActivityParticipationServiceImpl.registerActivity()` 使用 `LocalDate.now()` 写入 `activity_date`，而不是活动真实日期。
  - 验收标准：
    - 报名记录中的 `activity_date` 来自 `activity_schedule.activity_date`。
    - 新增单元测试覆盖“报名未来日期活动”的场景。
    - 历史数据提供一次性修复脚本（如需要）。

- [ ] **增加多模态上传防护（类型+大小+数量）**
  - 目的：防止大文件/异常文件导致内存压力和服务不可用。
  - 现状：`AIController.multimodalChat()` 仅校验媒体大类，`toNamedResource()` 会把文件一次性读入内存。
  - 验收标准：
    - 配置并启用单文件大小、总大小、最大文件数限制。
    - 对超限与不支持格式返回标准化错误响应。
    - 压测下服务不出现 OOM。

## P1（重要改进）

- [ ] **统一错误码与异常处理**
  - 目的：提升前后端联调效率与可观测性。
  - 现状：业务层大量返回字符串（如 `"Success"`、`"Failed..."`），缺少统一响应模型与全局异常处理。
  - 验收标准：
    - 引入统一返回体（code/message/data/requestId）。
    - 增加 `@RestControllerAdvice` 处理参数、业务、系统异常。
    - Controller/Tool 不再直接拼接错误字符串。

- [ ] **补齐测试体系（当前仅有 contextLoads）**
  - 目的：降低回归风险，支持持续迭代。
  - 现状：`src/test/java/cn/lc/sunnyside/SunnySideApplicationTests.java` 仅有启动测试。
  - 验收标准：
    - 为 `DailyTool` 和关键 Service 增加单元测试。
    - 为 `AIController` 增加接口测试（含参数缺失、异常、成功路径）。
    - 覆盖率达到可接受基线（例如行覆盖率 >= 60%，核心逻辑更高）。

- [ ] **修复老人匹配歧义与查询稳定性**
  - 目的：避免同姓/模糊匹配导致错误工具调用。
  - 现状：`ElderlyUserMapper.selectBySurname` 使用 `%LIKE%` 且接口返回单对象，存在多结果歧义风险。
  - 验收标准：
    - 返回候选列表并做 disambiguation（澄清提问）或明确唯一键策略。
    - 多同姓数据下调用不抛异常，且结果可解释。

- [ ] **优化“老人位置”能力为真实业务语义**
  - 目的：让工具返回可读、可执行的信息。
  - 现状：`ElderlyUserServiceImpl.getElderLocation()` 返回活动 DTO 列表的 `toString()`，不等于“位置”。
  - 验收标准：
    - 明确定义“位置”来源（实时定位/最近活动地点/默认房间）。
    - 返回结构化字段（location/sourceTime/confidence）。
    - 对无数据时返回可理解兜底文案。

- [ ] **日志脱敏与级别治理**
  - 目的：避免敏感信息落盘并减少生产噪音。
  - 现状：工具日志打印访客姓名、电话等；`application.yml` 对 `org.springframework.ai` 开启 DEBUG。
  - 验收标准：
    - 手机号、姓名等进行脱敏或不打印。
    - 生产环境关闭 DEBUG，按 profile 管理日志级别。
    - 补充关键链路日志字段（requestId/userId/toolName/latency）。

## P2（增强与体验优化）

- [ ] **对话记忆持久化与会话治理**
  - 目的：避免重启丢失上下文，支持多实例部署。
  - 现状：`AIconfig` 使用 `MessageWindowChatMemory` 内存实现。
  - 验收标准：
    - 引入持久化 ChatMemory（Redis/DB）。
    - 支持会话过期与手动清理。
    - 多实例下同会话记忆一致。

- [ ] **RAG 数据加载流程完善**
  - 目的：提升数据更新可控性与可观测性。
  - 现状：`RAGConfig` 启动时加载并写向量，依赖 `target/rag/*.sha256` 本地文件。
  - 验收标准：
    - 增加管理接口或任务触发机制（手动重建/增量更新）。
    - 引入加载耗时、分片数、失败重试指标。
    - 文档说明“开发/测试/生产”差异行为。

- [ ] **补充 API 文档与运行手册**
  - 目的：降低接入成本，便于协作交付。
  - 验收标准：
    - 增加接口文档（示例请求/响应、错误码、鉴权说明）。
    - README 补全本地启动、依赖服务（MySQL/Qdrant）、环境变量说明。
    - 增加常见故障排查（模型调用失败、向量库连接失败等）。

- [ ] **引入质量门禁（CI）**
  - 目的：把“可运行”提升为“可持续交付”。
  - 验收标准：
    - CI 至少包含编译、单测、基础静态检查。
    - 合并前自动执行并阻断失败变更。
    - 输出测试报告与覆盖率趋势。

## 功能优化建议（新增）

### P1（优先提升核心业务体验）

- [ ] **活动报名前冲突检测与候补机制**
  - 目的：避免老人重复报名同时间段活动，并提升活动满员时的可用性。
  - 验收标准：
    - 报名时校验同一老人在同时间段是否已有 `REGISTERED/ATTENDED` 活动。
    - 活动满员时支持进入候补队列并返回排队序号。
    - 取消报名后自动按顺序递补并通知用户。

- [ ] **来访预约审批流与通知闭环**
  - 目的：把当前“仅创建预约”升级为完整业务闭环，减少线下沟通成本。
  - 验收标准：
    - 新增预约审批动作（`APPROVED/REJECTED`）及备注字段。
    - 家属可查询预约进度与结果原因。
    - 审批结果可通过站内消息/短信（可选）通知。

- [ ] **老人健康提醒从固定文案升级为规则引擎**
  - 目的：根据老人个体差异输出更有价值的提醒。
  - 验收标准：
    - 基于老人标签（慢病、作息、饮食禁忌）生成差异化提醒。
    - 支持按时间段触发（早/中/晚）与事件触发（活动前后）。
    - 无个性化配置时回退通用提醒模板。

- [ ] **AI 工具调用增加“关键操作二次确认”**
  - 目的：降低误触发导致的错误报名、错误取消等风险。
  - 验收标准：
    - 对 `register/cancel/bookVisit` 等写操作先返回确认摘要。
    - 用户确认后再真正执行写入。
    - 审计日志记录“提议->确认->执行”全链路。

### P2（增强能力与家属协同）

- [ ] **家属视角查询能力（按老人聚合）**
  - 目的：让家属一次性看到活动、来访、公告、菜单，提升信息透明度。
  - 验收标准：
    - 提供“老人今日概览”接口（活动安排、预约状态、重要公告、当日菜单）。
    - 支持按日期查询历史概览。
    - 对外返回统一结构，便于前端卡片化展示。

- [ ] **公告智能分发与优先级提醒**
  - 目的：避免重要公告被淹没，提高触达率。
  - 验收标准：
    - 高优先级公告在 AI 回复中主动插入摘要提示。
    - 支持按人群标签（家属/老人/护理人员）筛选公告。
    - 过期公告自动降权或不再推送。

- [ ] **多模态能力落地到养老场景模板**
  - 目的：把“可上传媒体”变成“可直接解决问题”的功能。
  - 验收标准：
    - 提供至少 3 个预置场景：药盒识别、体检单解读、康复动作拍照反馈。
    - 每个场景有固定输出模板（结论、风险点、建议下一步）。
    - 无法识别时给出安全兜底建议并提示人工复核。

- [ ] **活动与探访的自然语言日程管理**
  - 目的：让用户用一句话完成查询与变更，降低操作门槛。
  - 验收标准：
    - 支持“帮我把明天下午探访改到三点”这类时间改期语义。
    - 对模糊时间（如“下周一上午”）能正确解析并回显确认。
    - 失败时返回可执行的替代建议（最近可用时段）。

## 建议排期（两周示例）

- 第 1 周：完成 P0（密钥、CORS、认证限流、报名日期修复、上传防护）。
- 第 2 周：完成 P1（统一异常、测试补齐、老人匹配/位置语义、日志治理）。
- P2 作为并行优化项穿插推进。
