# SQL 基线项目完善计划

## 1. 目标与边界
- 以 `sql.sql` 为唯一数据模型基线，统一后端实现与待办规划。
- 对 `todo.md` 中与 SQL 不一致或无字段支撑的功能进行“舍弃标注”，保留清单供你后续决策是否补模型。
- 在不新增 SQL 表结构的前提下，优先补齐当前 SQL 已定义但代码未落地的业务能力。

## 2. 现状基线结论（已核对）
- 已有业务表（活动、公告、老人、参与记录、值班、菜单、探访）已接入 Mapper/Service。
- `family_user`、`family_elder_relation`、`health_record` 已在 SQL 中定义，但当前无 Mapper/Service/Tool 落地。
- `todo.md` 中存在“新增上述三张表”的描述，与 SQL 现状冲突，需改为“完善实现”。
- `todo.md` 中若干能力缺乏 SQL 字段支撑（候补队列、预约拒绝原因、老人标签、消息推送、用药计划、SOS 事件等），需标注为“暂不实现（缺少 SQL 支撑）”。

## 3. 执行步骤

### 步骤 A：按 SQL 重构待办清单（先文档后代码）
- 生成“SQL 对齐版待办”：
  - 保留：可直接由现有 SQL 字段支撑的任务。
  - 舍弃并标注：当前 SQL 无法支持的任务，附“缺失字段/缺失表”说明。
  - 纠正：`todo.md` 中“新增 family_user/family_elder_relation/health_record”的冲突表述，改为“实现接入与能力落地”。
- 输出单独章节：`已舍弃功能（待你确认是否补充数据库设计）`，逐条列出建议新增的数据模型点。

### 步骤 B：补齐 SQL 已有但代码缺失的数据访问层
- 为 `family_user`、`family_elder_relation`、`health_record` 新增：
  - DO 实体；
  - Mapper 接口与 MyBatis XML；
  - 基础查询能力（按主键、按老人/家属关系、按日期区间等核心查询）。
- 复用现有命名风格、包结构与 MyBatis 映射规范，确保与现有模块一致。

### 步骤 C：补齐服务层与工具层可用能力（仅限 SQL 可支撑范围）
- 新增/扩展 Service：
  - 家属绑定老人关系校验；
  - 健康记录查询与汇总；
  - 家属视角的老人信息聚合查询（基于现有表可读信息）。
- 扩展 AI Tool 调用入口，提供可被对话链路触发的只读能力与必要写能力（仅覆盖 SQL 支撑的动作）。
- 明确不可做项：涉及候补队列、消息推送、SOS、用药计划等任务仅保留标注，不进入本次实现。

### 步骤 D：统一响应与错误处理的最小闭环（与 SQL 对齐改造配套）
- 在新增/改造接口链路中统一返回结构与异常映射，避免字符串式错误返回继续扩散。
- 对权限/绑定关系失败、数据不存在、参数非法等场景提供可区分的业务错误。

### 步骤 E：测试与回归验证
- 为新增 Mapper/Service 编写单元测试（重点覆盖关系校验、健康记录查询边界）。
- 为 AI 对话相关入口补充关键接口测试（成功、参数缺失、无权限、无数据）。
- 执行全量测试与编译检查，确保改造后主链路可运行。

## 4. 舍弃规则（实施时严格执行）
- 规则 1：`todo.md` 的需求只要依赖 SQL 中不存在的字段/表，本轮一律标记“舍弃（待设计）”。
- 规则 2：若需求可拆分为“SQL 可实现子集 + SQL 不可实现子集”，只实现可实现子集，并在待办中显式拆分标注。
- 规则 3：不隐式新增数据库结构；任何新增模型建议仅写入“待你确认”清单，不直接落库。
